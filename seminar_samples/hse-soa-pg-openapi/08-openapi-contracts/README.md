# 08 — OpenAPI Контракты для Нескольких Сервисов

## Концепция API-First разработки

**API-First** — подход, при котором сначала проектируют API (контракт),
а затем пишут реализацию.

### Преимущества
1. **Независимая разработка**: фронтенд и бэкенд команды работают параллельно
2. **Чёткий контракт**: все знают что именно возвращает каждый эндпоинт
3. **Генерация кода**: можно автоматически генерировать клиенты на любом языке
4. **Моки**: можно запустить mock-сервер из спецификации пока реализация не готова

## Архитектура примера

```
┌─────────────────────┐    REST     ┌─────────────────────┐
│   users-service     │◄───────────►│   orders-service    │
│   (Go)              │             │   (Python)          │
│   /users            │             │   /orders           │
└─────────────────────┘             └─────────────────────┘
         ↑                                    ↑
    api/users.yaml                     api/orders.yaml
    (контракт)                         (контракт)
```

Обратите внимание: orders-service вызывает users-service для валидации `user_id`.
Это межсервисное взаимодействие — ключевой паттерн микросервисов.

## Версионирование API

Важный вопрос: как изменять API не сломав существующих клиентов?

### Версия в URL (наиболее распространённо)
```
/v1/users
/v2/users
```

### Версия в заголовке
```
Accept: application/vnd.myapi.v2+json
```

### Правила обратной совместимости
**Не ломающие изменения (можно в рамках версии):**
- Добавить новое необязательное поле в ответ
- Добавить новый эндпоинт
- Добавить новый необязательный query параметр

**Ломающие изменения (требуют новой версии):**
- Удалить поле из ответа
- Изменить тип поля
- Сделать опциональное поле обязательным
- Изменить семантику эндпоинта

## Swagger UI для нескольких API

В этом примере Swagger UI настроен для показа нескольких спецификаций:

```yaml
environment:
  SWAGGER_JSON_URL: /specs/users.yaml
  # Или можно показывать несколько через URLS:
  URLS: "[{url: '/specs/users.yaml', name: 'Users API'}, {url: '/specs/orders.yaml', name: 'Orders API'}]"
```

## Запуск

```bash
docker compose up -d
```

Открой http://localhost:8080 — Swagger UI с возможностью переключаться между API.

Переключатель в правом верхнем углу позволяет выбирать между:
- **Users API** — управление пользователями
- **Orders API** — управление заказами

## Изучи контракты

Открой `api/users.yaml` и `api/orders.yaml` — обрати внимание на:

1. Как описан `user_id` в заказах — ссылка на пользователя из другого сервиса
2. Как описаны ошибки: единый формат `ErrorResponse` в обоих API
3. Как используются `$ref` для переиспользования схем
4. Как описан список статусов заказа через `enum`
