# Архитектура маркетплейса

## Задание 3: Домены и их ответственность

### Перечень доменов

| Домен | Описание ответственности |
|-------|--------------------------|
| **Пользователи** | Управление профилями покупателей и продавцов, регистрация, аутентификация, верификация продавцов, восстановление пароля, управление ролями (покупатель/продавец/админ) |
| **Каталог** | Управление товарами, категориями, поиском и фильтрацией. Хранение информации о товарах: описания, цены, изображения, остатки на складе, рейтинги и отзывы |
| **Заказы** | Управление корзиной, создание и обработка заказов, отслеживание статусов (новый → оплачен → в сборке → в доставке → завершён). Обработка возвратов и отмен заказов |
| **Уведомления** | Отправка уведомлений пользователям о статусах заказов (подтверждение, оплата, доставка), акциях и новостях. Поддержка email-уведомлений |
| **Оплата** | Обработка платежных транзакций, интеграция с внешними платёжными системами (эквайринг), расчёт и учёт платежей |

## Задание 4: Распределение доменов по сервисам

### Сервисы и их домены

| Сервис | Домен | Логика разбиения |
|--------|-------|------------------|
| **API Gateway** | — | Единая точка входа для всех запросов. Централизованная аутентификация, маршрутизация, ограничение скорости (rate limiting) и агрегация ответов от микросервисов |
| **Менеджер пользователей** | Пользователи | Управление пользователями — отдельная бизнес-область с уникальными правилами для покупателей и продавцов. Продавцы проходят верификацию, имеют расширенные профили и права. Выделение позволяет независимо масштабировать аутентификацию |
| **Каталог** | Каталог | Товарный каталог — ядро маркетплейса, высоко нагруженная часть с частыми запросами на чтение. Выделение позволяет оптимизировать поиск, кэширование и масштабировать независимо от остальных сервисов |
| **Сервис заказов** | Заказы | Заказы имеют чёткий жизненный цикл с транзакционными операциями. Выделение в отдельный сервис позволяет управлять состоянием заказов независимо от каталога и пользователей, обеспечивая целостность данных |
| **Сервис оповещений** | Уведомления | Уведомления — вспомогательная функция, которая может работать асинхронно и не критична для основного потока заказов. Выделение позволяет масштабировать отправку сообщений отдельно и легко добавлять новые каналы (SMS, push) |
| **Система оплаты** | Оплата | Платежи — критически важная область с требованиями к безопасности и надёжности. Выделение в отдельный внешний сервис позволяет изолировать финансовую логику и интеграции с банками |

## Задание 5: Границы владения данными

### Базы данных и их владельцы

| База данных | Владелец | Данные | Ответственность |
|-------------|----------|--------|-----------------|
| **БД пользователей** | Менеджер пользователей | Профили покупателей и продавцов, учётные данные (хеши паролей), адреса доставки, история активности, сессии, права доступа | Хранение и управление всеми данными о пользователях. Только менеджер пользователей имеет право на запись |
| **БД товаров** | Каталог | Товары, категории, описания, цены, изображения, остатки на складе, рейтинги, отзывы | Хранение всего товарного каталога. Только каталог имеет право на запись, сервис ленты может читать для формирования персонализированной выдачи |
| **БД заказов** | Сервис заказов | Заказы, корзины, статусы, история изменений, связь с пользователями и товарами | Хранение всех данных о заказах и их жизненном цикле. Только сервис заказов имеет право на запись |
| **Внешняя платёжная система** | Система оплаты | Транзакции, история платежей, статусы оплаты | Хранение финансовых данных. Интеграция через внешний сервис для обеспечения безопасности |
| **Провайдер электронной почты** | Сервис оповещений | Шаблоны сообщений, история отправок | Хранение данных об уведомлениях. Интеграция через внешний сервис для массовой рассылки |

### Взаимодействия сервисов

#### Синхронные вызовы (HTTP, ожидание ответа)

| От | К | Описание |
|----|---|----------|
| Веб-приложение | API Gateway | Все запросы от фронтенда проходят через единый шлюз |
| API Gateway | Каталог | Запросы на поиск и просмотр товаров |
| API Gateway | Сервис заказов | Запросы на создание и просмотр заказов |
| API Gateway | Менеджер пользователей | Запросы на аутентификацию и управление профилем |
| Каталог | БД товаров | Чтение/запись данных о товарах и категориях |
| Сервис заказов | БД заказов | Чтение/запись данных о заказах и корзинах |
| Менеджер пользователей | БД пользователей | Чтение/запись данных о пользователях и сессиях |
| Сервис заказов | Система оплаты | Инициация платежа и проверка статуса оплаты |

#### Асинхронные вызовы (без ожидания ответа)

| От | К | Описание |
|----|---|----------|
| Сервис заказов | Сервис оповещений | Отправка уведомления о создании/изменении заказа |
| Каталог | Сервис оповещений | Отправка уведомления о новых товарах от избранных продавцов |
| Сервис оповещений | Провайдер почты | Отправка электронных писем пользователям |

### Принципы владения данными

1. **Каждая база данных имеет единственного владельца** — только один сервис имеет право на запись в каждую БД
2. **Чтение разрешено только через владельца** — другие сервисы получают данные через вызовы API, а не напрямую к БД
3. **Нет разделяемых баз данных** — исключение: кэш (если используется), но это не основное хранилище
4. **Внешние системы** — платёжная система и почтовый провайдер являются внешними зависимостями, данные в них управляются через соответствующие сервисы

### Задание 7: Trade-off'ы вариантов

#### Вариант А: Микросервисы

| Категория | Детали |
|-----------|--------|
| **Плюсы** | ✅ Независимое масштабирование (каталог можно масштабировать отдельно от заказов)<br>✅ Изоляция сбоев (падение оповещений не ломает оформление заказов)<br>✅ Параллельная разработка (команда каталога не мешает команде заказов)<br>✅ Гибкость технологий (каталог на Go, рекомендации на Python) |
| **Минусы** | ❌ Сложнее локальная разработка (нужно запускать несколько сервисов)<br>❌ Сетевые вызовы добавляют латентность (~50-100 мс на вызов)<br>❌ Распределённые транзакции сложнее (нужен паттерн Saga)<br>❌ Выше накладные расходы на инфраструктуру (мониторинг, логирование) |
| **Trade-off** | **Масштабируемость vs Сложность**<br>Микросервисы дают возможность точно масштабировать нагруженные части (каталог во время распродаж), но требуют больше усилий на инфраструктуру и отладку. Для маркетплейса с неравномерной нагрузкой (каталог — читает много, заказы — пишет редко) этот trade-off оправдан. |

#### Вариант Б: Монолит

| Категория | Детали |
|-----------|--------|
| **Плюсы** | ✅ Простота разработки (один репозиторий, один деплой)<br>✅ Быстрая отладка (всё в одном процессе)<br>✅ Простые транзакции (одна БД = одна транзакция)<br>✅ Нет сетевых вызовов внутри приложения (выше производительность) |
| **Минусы** | ❌ Невозможно масштабировать отдельные части (при нагрузке на каталог масштабируем ВСЁ)<br>❌ Единая точка отказа (ошибка в оповещениях может уронить весь маркетплейс)<br>❌ Конфликты в команде (несколько разработчиков правят один код)<br>❌ Рост технического долга («Большой шар грязи» после 50к строк) |
| **Trade-off** | **Простота vs Масштабируемость**<br>Монолит проще запустить и поддерживать на старте, но становится узким местом при росте. Для маркетплейса с ожидаемым ростом пользователей и товаров этот trade-off неприемлем — рано или поздно придётся рефакторить в микросервисы с потерей времени и ресурсов. |

### Задание 8: Обоснование финального выбора

#### Выбрана архитектура: **Микросервисы по доменам (Вариант А)**

#### Аргументация на основе требований кейса

| Требование кейса | Почему микросервисы подходят лучше |
|------------------|-----------------------------------|
| **Персонализированная лента товаров** | Требует анализа поведения пользователей и тяжёлых вычислений. В микросервисах выделяется отдельный сервис `Лента`, который можно масштабировать независимо от каталога и не нагружать основные операции. |
| **Управление товарным каталогом (для продавцов)** | Продавцы работают только с каталогом. В микросервисах каталог изолирован — ошибки продавцов не влияют на оформление заказов покупателями. |
| **Оформление заказов** | Заказы имеют строгий жизненный цикл с транзакционными операциями. В микросервисах сервис заказов управляет своей БД, гарантируя целостность данных без влияния на каталог. |
| **Расчёт и учёт платежей** | Платежи — критически важная область с требованиями к безопасности. В микросервисах платёжный сервис изолирован, имеет отдельную инфраструктуру и аудит, что соответствует требованиям финансовой безопасности. |
| **Отправка уведомлений** | Уведомления — вспомогательная функция, которая может работать асинхронно. В микросервисах сервис оповещений не блокирует основные операции (например, если почтовый провайдер недоступен, заказы продолжают оформляться). |

#### Почему монолит отклонён

Монолитная архитектура **противоречит ключевым требованиям маркетплейса**:

1. **Невозможность независимого масштабирования**  
   При распродаже нагрузка падает на каталог, но в монолите придётся масштабировать ВСЁ приложение (включая оповещения и пользователей), что неэффективно по ресурсам.

2. **Риск единой точки отказа**  
   Ошибка в модуле оповещений (например, зависание при отправке email) может заблокировать оформление заказов — критическая проблема для маркетплейса.

3. **Конфликт интересов продавцов и покупателей**  
   В монолите код для продавцов (управление каталогом) и покупателей (оформление заказов) находится в одном месте. Это усложняет развитие функционала для каждой группы независимо.

#### Компромиссы и их управление

| Сложность микросервисов | Как мы её решаем |
|-------------------------|------------------|
| Сетевые вызовы | Кэширование через Redis для часто запрашиваемых данных (товары, профили) |
| Распределённые транзакции | Паттерн Saga для заказов: «зарезервировать товар → создать заказ → списать деньги» с откатом при ошибке |
| Сложность локальной разработки | Docker Compose для запуска всех сервисов одной командой `docker-compose up` |

#### Вывод

Микросервисная архитектура выбрана, потому что **требования кейса маркетплейса напрямую выигрывают от независимости компонентов**:
- Каталог и заказы имеют разную нагрузку → независимое масштабирование критично
- Платежи требуют изоляции → безопасность выше в отдельном сервисе
- Оповещения не должны блокировать основные операции → асинхронность решает проблему

Монолит упростил бы старт, но создал бы архитектурные проблемы уже на этапе роста пользовательской базы — что неизбежно для маркетплейса. Поэтому микросервисы — **оправданный выбор с учётом долгосрочной перспективы**.
